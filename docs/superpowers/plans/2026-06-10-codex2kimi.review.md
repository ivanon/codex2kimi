## 三、结构与术语问题

### 1. 文件命名不一致

- **File Structure 表**（L46）写的是 `deploy/com.codex2kimi.proxy.plist`
- **Task 14** 实际创建的是 `deploy/com.codex2kimi.proxy.plist.template`

建议统一为 `.plist.template`，并在 File Structure 表中同步修正，避免 agent 创建错误文件名。

### 2. 入口文件冗余

- Task 1 创建 `src/index.ts`（占位 + smoke test）
- Task 12 起实际入口为 `src/server.ts`（`dev`/`start` 均指向 server）

`src/index.ts` 在 Task 12 之后成为 dead code。建议：Task 12 完成后删除 `src/index.ts`，或将 smoke test 改为 import `VERSION` 从 `server.ts` / 独立 `src/version.ts`。

### 3. Runbook 条目数不一致

Task 13 runbook 标题写「手动五条路径」，实际列出 **6** 条（含流式中断）。建议改为「六条路径」或合并条目。

### 4. tsconfig 与 Node ESM

`moduleResolution: "Bundler"` 配合 `.js` 扩展名 import 在 tsx/vitest 下通常可用，但对纯 `tsc` + `node dist/` 产物，更稳妥的是 `"moduleResolution": "NodeNext"` + `"module": "NodeNext"`。当前配置不是 blocker，但 implementer 若遇到类型解析问题应优先检查此项。

## 四、Proposed Code 评审

### 高优先级（建议实现前修正计划）

#### 1. `pumpStream` 错误分类与设计不符

设计 spec 明确：

- JSON 解析失败 / 上游 4xx/5xx → `response.failed`
- 中途断流 / Codex abort → `response.incomplete`

计划中的 `pumpStream`（Task 12）在 `catch` 中**一律**调用 `translator.incomplete()`：

```typescript
} catch (err) {
  emit(translator.incomplete());
}
```

问题：`JSON.parse(frame.data)` 失败、畸形 SSE 等应走 `translator.fail(...)` 或等价逻辑，而非 `incomplete`。

**建议**：在计划中明确分支——`AbortError` → `incomplete()`；parse/逻辑错误 → `fail()`；并在 `tests/server.test.ts` 增加对应断言。

#### 2. 流式 `error` 事件未走 `error.ts` 映射

Task 10 中 `StreamTranslator.handle` 对 Anthropic `error` 事件直接透传 `event.error.type`（如 `overloaded_error`）到 `response.failed`，而 Task 8 的 `TYPE_MAP` 会把 `overloaded_error` 映射为 `api_error`。

非流式路径经 `translateError`，流式路径不经映射——**行为不一致**，Codex 端可能因 error type 不识别而表现异常。

**建议**：流式 error 分支复用 `TYPE_MAP`，或在计划中注明「流式 failed 保留 Anthropic 原始 type」并给出 design 决策依据。

#### 3. 空 `messages` / 无效 input 未校验

`translateRequest` 可能产出 `messages: []`（例如 input 仅含 system/developer/reasoning items）。Anthropic API 要求至少一条 user/assistant message，会直接 400。

计划仅校验 `Array.isArray(body.input)`，未校验翻译后 messages 非空。

**建议**：在 Task 12 server 路由中，`translateRequest` 后若 `messages.length === 0`，返回 `badRequest("input must contain at least one user or assistant message")`；补单测。

#### 4. `tool_choice: "none"` 与 tools 共存

设计 spec：`none` → 「不带 tools 或 `{type:"none"}`」。计划 Task 6 映射为 `{type:"none"}` 但**仍保留** `out.tools`。部分 Anthropic 兼容端会拒绝此组合。

**建议**：当 `tool_choice === "none"` 时，计划中明确删除 `out.tools`（或在联调 Task 13 能力清单中标注待验证）。

#### 5. 端口配置无校验

`resolveConfig` 中 `Number(portRaw)` 对非法 env（如 `CODEX2KIMI_PORT=abc`）会得到 `NaN`，服务可能以异常方式启动。

**建议**：Task 3 增加 port 范围校验（1–65535）及 `ConfigError` 测试。

### 中优先级（实现时注意）

#### 6. SSE 解析器边界

Task 4 的 `parseSSEStream`：

- 未处理 `\r\n` 行尾（真实 SSE 常见 CRLF）
- 未忽略 SSE 注释行（`: keep-alive`）
- Kimi 若发送 `event:` 无 `data:` 的 ping 帧，当前会 yield `{ event, data: "" }`，`pumpStream` 中 `if (!frame.data) continue` 会跳过——行为正确，但建议在 sse 单测中加 CRLF case

#### 7. `parseArgs` 静默吞掉非法 JSON

Task 5 对 malformed `function_call.arguments` 返回 `{}`，上游 tool_use 会以空 input 发出，调试困难且可能导致工具链 silent failure。

**建议**：至少保留 raw string 作为 fallback，或在 v1 文档中标注为 known limitation。

#### 8. 非流式响应：连续 text 块未合并

Task 7 注释已说明「每块一个 message item」，与设计 spec「连续 text 块合并进同一个 message output item」的表述略有偏差。若 Codex 非流式路径使用频率低，可接受；若联调发现问题，合并逻辑应在 `response.ts` 调整——建议在 Task 13 runbook 加一条非流式多 text block 验证。

#### 9. `usage` 字段不完整

- 非流式：`cache_creation_input_tokens` 未映射到 Responses usage
- 流式：`reasoning_tokens` 未映射（spec 类型已预留 `output_tokens_details.reasoning_tokens`）

v1 可 defer，但应在已知限制中列出。

#### 10. 上游超时与客户端断开

- `callUpstream` 无默认超时（spec 建议 120s）
- Codex 断开时虽透传 `c.req.raw.signal`，但 `pumpStream` 内部未将 signal 传给 upstream stream reader；需在 Task 12 注明：`ReadableStream` cancel 时是否联动 upstream abort（当前计划未覆盖）

#### 11. `StreamTranslator.fail()` 未被 server 使用

Task 10 实现了 `fail()`，Task 12 `pumpStream` 未调用。与第 1 点一并修正。

#### 12. `main()` 启动检测

`process.argv[1].endsWith("server.js")` 对 `node dist/server.js` 有效；`tsx watch src/server.ts` 不经过此分支（符合预期）。若未来用 `node --import tsx src/server.ts` 等路径，可能误触发或不触发——建议在 README 明确「生产入口仅 `dist/server.js`」。

### 低优先级 / 优化

- Task 1 smoke test 依赖 `src/index.ts`，与最终架构略脱节
- 计划每 Task 要求 commit；若由人工驱动开发，可按 milestone 合并 commit
- `install.sh` 未检查 `~/.config/codex2kimi/config.json` 是否存在或权限是否为 600（README 有说明，脚本可加 warn）
- replay 测试只断言 substring 包含，未验证 `sequence_number` 连续性或 JSON schema——对回归足够，可增强
